## Operating rules
- **Newest entries must always be added at the top** of each section so an agent does not need to read the whole file.
- Keep entries short, concrete, and action-oriented.
- If an item is no longer relevant, change its status instead of silently deleting it unless it is pure noise.
- Agents should read this file **before** starting or resuming unattended work.
- When an item here affects execution truth, the agent should reflect that in `todo.md` and/or `NIGHT_RUN_REPORT.md` as appropriate.
- If a human reports an external fix or settings change, the previous blocker must be treated as **pending revalidation** until a fresh smoke/runtime verification confirms `resolved` or `blocked-external` again.
- When an agent consumes a human note that changes blocker truth, it must add a short entry to the Agent consumption log with what was read, what was rechecked, and what status was concluded.
- **Recurrent unattended-run rule:** if the latest state is genuinely waiting on external human action, keep it blocked. But if the latest state is only waiting because docs/tracker/control-plane truth is stale or incomplete, the agent should determine the next concrete action itself and proceed immediately (doc sync, revalidation, implementation, QA, handoff, or next-feature preparation) without asking for extra human sign-off.
- **Ambition read rule:** before next-feature selection, doc shaping, or autonomous feature advancement, read `docs/product/AMBITION.md` and use it as the strategic north star for deciding what kind of progress is high-leverage versus merely easy.
- **NextAction rule:** after reading BLACKBOARD + current package tracker, the agent must classify the current state as one of: `blocked-external`, `pending-revalidation`, `ready-docs`, `ready-implementation`, `ready-qa`, `ready-handoff`, or `ready-next-feature`. If not `blocked-external`, the agent should choose and execute the next highest-leverage action instead of stopping at a passive waiting state.
- **No-sign-off advancement rule:** explicit human sign-off is preferred when available, but it is no longer a mandatory gate for feature advancement. If the current package is truthfully complete or blocked on a real external dependency, and the next safe package is already sequenced or can be selected from existing BLACKBOARD guidance, the agent should advance autonomously into next-feature preparation and then into implementation when the docpack is sufficiently locked.
- **tmux liveness rule:** the existence of a tmux session alone does **not** prove that a worker is active. Treat a worker as truly active only if recent pane output or process state shows ongoing execution and no completion/stop-point state has already been reached. A stale shell, idle prompt, or post-run tmux session must not block a new NextAction.
- **Worker-finish rule:** when a worker reaches a truthful completion or stop point, the agent should prefer ending the shell cleanly (or explicitly marking the tmux session stale) so later orchestrators do not misclassify leftover tmux sessions as live work.
- **Coding-agent dispatch rule:** when implementation work is the next action, use the OpenClaw toolchain and explicitly prefer an OpenCode `/ulw-loop` run inside `tmux` for unattended coding execution, unless a newer repo-specific instruction overrides it.
- **Autonomous execution policy:** default to `act-first, report-second`. If the active package is incomplete, the next documented task is in scope, and no true external blocker is active, the agent should execute/resume the next action without waiting for fresh human confirmation.
- **Auto-relaunch rule:** if a recurrent or resumed check finds that no worker is truly active and the current package remains incomplete with a documented, safe next action, automatically launch a new worker instead of sending a status-only update.
- **Status-only exception rule:** send a status-only update without relaunch only when (1) a true human decision is required that cannot be safely inferred from existing docs/control-plane truth, (2) a real external blocker is active, or (3) repeated execution failed and now needs intervention. Package completion alone is not sufficient reason to stop if the next feature can be prepared or advanced safely.
- **Repeated-failure freeze rule:** if the same active package reproduces materially the same runtime/auth failure across multiple fresh revalidation attempts and no new implementation step becomes truthful, freeze that package for automatic closeout relaunches. Preserve it as `intervention-needed`, record the best current diagnosis, and allow next-feature preparation to continue in parallel rather than burning unattended cycles on blind repetition.
- **Execution-vs-preparation split rule:** when a frozen or intervention-needed package still has unresolved runtime truth, BLACKBOARD may keep that package as the active execution package while separately naming an active preparation package for next-feature doc shaping. Do not force all forward progress to wait on the frozen package unless a newer human note explicitly requires that.
- **Revalidation-before-escalation rule:** if an external or provider-side issue may have changed, downgrade the old blocker to `pending-revalidation`, run the smallest truth-finding smoke/recheck first, and only then decide whether to relaunch, continue, or escalate.
- **Stop-point discipline rule:** do not stop only to summarize progress if more in-scope work remains and no human decision is required. Before any stop, sync BLACKBOARD + tracker/report truth and either continue work or record exactly why autonomous continuation is not allowed.
- **Feature-chain continuity rule:** when the current package reaches truthful closeout or a durable external blocker, the agent should immediately ask "what is the safest next leverage step?" Use `docs/product/AMBITION.md` together with BLACKBOARD sequencing to prefer next packages that strengthen the core parent loop and avoid anti-goal drift. If a next package can be shaped, locked, or started from existing repo docs and BLACKBOARD guidance, do that work instead of waiting for ceremonial handoff.
- **Doc-lock sufficiency rule:** when the next package docpack is specific enough that an unattended worker can execute without guessing, treat that as sufficient authorization to begin implementation unless BLACKBOARD explicitly marks the package as waiting for human approval.
- **Prompt-thin rule:** recurrent prompts should remain thin and defer mission selection to BLACKBOARD, but BLACKBOARD itself must contain enough autonomy policy that a resumed agent can decide whether to relaunch a worker without guessing.
- **Prompt reference rule:** the recommended thin recurrent operator prompt for this repo is documented in `docs/guides/RECURRENT_UNATTENDED_PROMPT.md`. Treat that file as the default prompt template unless a newer human instruction overrides it.
- **QA execution rule:** when the next action is deterministic verification/QA (especially browser-driven protected-flow checks), prefer the most observable and reproducible execution path. `tmux` is allowed, but a direct `process/background` run with explicit logs, exit code, and artifact output is preferred whenever it gives better observability than an interactive worker shell.
- **BLACKBOARD writing rule:** prefer putting enduring operating logic, mission-selection rules, worker-liveness interpretation, QA account truth, and resumable NextAction guidance here rather than in cron prompts. Recurrent prompts should stay thin and defer to this file.
- **BLACKBOARD writing style rule:** write so a future agent can answer four questions quickly without reading the whole repo: (1) what package is active, (2) what is actually blocked vs merely stale, (3) what is the exact next action, and (4) how to tell whether a worker is really still alive.
- **BLACKBOARD hygiene rule:** when a new note supersedes older truth, either mark the older note `applied`, `resolved`, or `stale`, or add an explicit newer-top note that tells the agent how to interpret the older entry. Avoid leaving unresolved ambiguous notes that force the agent to guess which truth is current.
- **Snapshot-first maintenance rule:** when execution truth changes, update the `Current control-plane snapshot` first before expanding logs/history, so the next unattended agent can read current mission truth without reconstructing it from older sections.
- **Companion-doc sync rule:** if `todo.md` or `NIGHT_RUN_REPORT.md` still use older wording, treat the BLACKBOARD snapshot as current mission truth and sync the companion docs as soon as practical so historical wording does not masquerade as the current state.
